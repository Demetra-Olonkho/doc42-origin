## 1. Источники и авторы — такие же сущности, а не служебные метаданные

Во многих системах делают так:

```text
Character
Location
Event
```

— это сущности.

А вот:

```text
Автор
Книга
Источник
Редакция
Издание
```

— просто набор полей в БД.

Для вашего случая я бы так не делал.

Потому что очень быстро возникают вопросы:

```text
Кто впервые описал персонажа?

В каком произведении впервые появился город?

Какие источники противоречат друг другу?

Какая версия статьи является актуальной?

Какие материалы относятся к канону?
```

Это уже полноценные связи в графе знаний.

Например:

```text
Character
    │
    ├── first_appeared_in
    ▼
Document

Document
    │
    ├── created_by
    ▼
Author
```

или:

```text
Event
    │
    ├── described_in
    ▼
Source
```

---

## 2. Я бы даже разделил граф на два слоя

Условно:

```text
Lore Layer
```

и

```text
Knowledge Layer
```

### Lore Layer

Содержит внутримировые сущности.

```text
Character
Location
Organization
Event
Item
Species
Culture
Religion
```

---

### Knowledge Layer

Содержит сущности, описывающие знания о мире.

```text
Author
Source
Document
Edition
Publication
Contributor
Revision
```

---

Тогда можно строить связи между слоями.

Например:

```text
Author
    │
created
    ▼
Source

Source
    │
describes
    ▼
Character
```

---

## 3. Для Publishing Pipeline это действительно критично

Потому что тогда автоматически становятся возможны:

### Индексы источников

Это уже начинает напоминать академические системы цитирования.

## 5. Следствие: нельзя зашивать типы сущностей в код

Вот здесь я бы уже сейчас заложил очень важное требование.

Плохо:

```text
Entity
 ├─ Character
 ├─ Location
 ├─ Item
```

жёстко прописаны в программе.

---

Лучше:

```text
Entity Type Definition
```

как данные.

Например:

```yaml
entity_type:
  name: Character

fields:
  - name
  - biography
  - birth_date
```

---

Тогда пользователь сможет создать:

```text
Starship
Magic School
Noble House
Corporation
AI Network
Planetary System
```

без изменения исходного кода.

## 6. Мне кажется, сейчас начинает вырисовываться настоящая граница проекта

На данный момент я бы сформулировал миссию системы примерно так:

> Универсальная платформа управления знаниями о вымышленных мирах, поддерживающая хранение сущностей, отношений, источников, авторства, версионирования и публикации данных через различные внешние системы.

Обратите внимание, что в этом определении нет ни слова про:

- PostgreSQL;

- YAML;

- VTT;

- LaTeX;

- PDF.

Все они становятся лишь адаптерами и потребителями данных.

И это хороший признак. Обычно именно такие системы оказываются наиболее жизнеспособными в долгосрочной перспективе, потому что их ядро описывает предметную область, а не конкретную технологию или сценарий использования.

Если двигаться дальше, то следующим крупным вопросом я бы поставил уже не список сущностей, а вопрос **границ расширяемости**: что в системе должно быть фиксировано ядром, а что пользователь должен иметь возможность определять самостоятельно через схемы, плагины и расширения. Именно это обычно определяет, получится универсальная платформа или специализированный инструмент для одного класса сеттингов.
