> Какие понятия должны существовать в любой инсталляции Lore Database независимо от жанра, сеттинга, хранилища и интеграций?

# Уровень 0. Абсолютное ядро

Это те сущности, без которых система вообще перестаёт быть Lore Database.

```text
Core
├── Entity
├── Entity Type
├── Relation
├── Relation Type
├── Attribute
├── Source
├── Revision
└── Collection
```

---

## Entity

Базовый объект знаний.

Примеры:

```text
Персонаж
Город
Книга
Автор
Королевство
Орден
Планета
```

Для ядра всё это просто:

```text
Entity
```

---

Минимально:

```text
ID
Type
Attributes
Metadata
```

---

## Entity Type

Описание класса сущностей.

Например:

```text
Character
Location
Organization
```

Но это уже данные, а не код.

---

## Relation

Экземпляр связи.

Например:

```text
Альберт
  member_of
Гильдия торговцев
```

---

Минимально:

```text
Source Entity
Relation Type
Target Entity
```

---

## Relation Type

Описание допустимого типа связи.

Например:

```text
member_of
owns
located_in
```

---

Опять же:

данные,

а не код.

---

## Attribute

Значение свойства сущности.

Например:

```text
name
population
birth_date
```

---

Здесь я бы сразу заложил мысль:

```text
Attribute
=
отдельная сущность модели
```

а не просто поле таблицы.

Это сильно поможет в будущем.

---

## Source

Источник знания.

Например:

```text
Книга
Газеттир
Приключение
Черновик
Интервью автора
```

---

Ключевая идея:

Любой факт может иметь источник.

---

## Revision

Состояние объекта во времени.

Например:

```text
Версия мира 1.0

Версия мира 2.0
```

или:

```text
До войны

После войны
```

---

## Collection

Логическая группировка.

Например:

```text
Кампания

Линейка книг

Серия статей

Набор материалов
```

---

Очень полезно для публикаций и экспорта.

---

# Уровень 1. Метамодель

Это уже описание самой модели данных.

---

## Schema

Описание структуры.

Например:

```text
Character
```

содержит:

```text
name
birth_date
biography
```

---

## Field Definition

Описание поля.

Например:

```text
population
```

тип:

```text
integer
```

---

## Validation Rule

Правила проверки.

Например:

```text
population >= 0
```

---

или:

```text
birth_date < death_date
```

---

# Уровень 2. Происхождение знаний

Мне кажется, это одна из сильнейших сторон будущей системы.

---

## Agent

Источник создания знаний.

Примеры:

```text
Автор

Редактор

Издательство

Организация
```

---

Важно:

это не пользователь системы.

Это участник происхождения знания.

---

## Citation

Ссылка на источник.

Например:

```text
Факт X
основан на
Документе Y
```

---

## Assertion

Вот здесь появляется очень интересная сущность.

Не факт.

А утверждение.

Например:

```text
Король погиб в 721 году.
```

---

Это утверждение может:

```text
быть подтверждено;

быть опровергнуто;

быть спорным;

быть альтернативным.
```

---

Для сложных сеттингов это огромный плюс.

---

# Уровень 3. Таксономия

Например:


## Taxon

Например:

---

Это позволит не плодить жёсткие перечисления в коде.

# Что мне сейчас кажется самым важным

После всех обсуждений я бы сформулировал фундаментальную архитектурную аксиому проекта так:

```text
Lore Database хранит не сущности мира.

Lore Database хранит знания о сущностях мира.
```

Это небольшая формулировка, но последствия у неё огромные.

Потому что тогда естественным образом появляются:

- источники;

- авторство;

- версии;

- канон;

- альтернативные версии событий;

- противоречивые сведения;

- ссылки между публикациями;

- редакции;

- происхождение данных.

Именно это отличает будущую систему от обычной вики, обычной БД персонажей или набора YAML-файлов. Она начинает превращаться в полноценную платформу управления знаниями о вымышленных мирах.
