Тогда я бы предложил сделать следующий шаг и попробовать сформулировать **архитурную конституцию проекта** — набор принципов, которые впоследствии будут влиять на все решения по реализации.

Сейчас у нас уже наметилось несколько таких принципов.

---

# Принцип 1. Lore Core первичен

Не:

```text
VTT → использует данные
```

и не:

```text
Publishing Pipeline → использует данные
```

а:

```text
Lore Core
    │
    ├── VTT
    ├── Publishing Pipeline
    ├── Wiki
    ├── API
    └── другие потребители
```

То есть ядро знаний мира является самостоятельным продуктом.

---

# Принцип 2. Модель важнее способа хранения

Не:

```text
PostgreSQL-модель
```

а:

```text
Lore-модель
```

которая может храниться через:

```text
YAML
JSON
SQLite
PostgreSQL
REST
Graph DB
```

Это избавляет нас от привязки к конкретной технологии.

---

# Принцип 3. Все объекты являются сущностями

Не только:

```text
Character
Location
Event
```

но и:

```text
Book
Gazetteer
Campaign
Author
Publisher
Edition
Source
```

---

То есть:

```text
Всё является сущностью.
```

Различаются лишь типы сущностей.

---

# Принцип 4. Все связи являются данными

Не:

```ada
Character.Organization
```

зашитое в код.

А:

```text
Character
     │
 member_of
     │
Organization
```

как запись в графе знаний.

---

# Принцип 5. Типы сущностей расширяемы

Ядро не знает что такое:

```text
Character
Starship
Spell
Corporation
```

Оно знает только:

```text
Entity Type
```

---

Пользователь определяет собственные типы.

---

# Принцип 6. Типы связей расширяемы

Не фиксированный список:

```text
owns
member_of
located_in
```

а пользовательские определения.

---

# Принцип 7. Документы — граждане первого класса

Это, как мне кажется, одна из самых сильных идей проекта.

Большинство систем делают так:

```text
Данные
↓
Документ
```

У нас же:

```text
Данные ↔ Документ
```

---

Книга, статья, газеттир, приключение и энциклопедия существуют в том же графе знаний.

---

# Принцип 8. Канон и источники поддерживаются ядром

Для миров это чрезвычайно важно.

Например:

```text
Источник А
утверждает X

Источник Б
утверждает Y
```

Система не обязана считать это ошибкой.

Она должна уметь хранить оба утверждения.

---

Появляется понятие:

```text
Canon Layer
```

или:

```text
Knowledge Provenance
```

(происхождение знаний).

---

# Принцип 9. Версионирование является частью модели

Не только Git.

А именно внутри модели знаний.

Например:

```text
Королевство до войны

Королевство после войны
```

---

или:

```text
Редакция 1
Редакция 2
Редакция 3
```

---

Это разные состояния знаний.

---

# Принцип 10. Платформа должна быть полезна без VTT и без Publishing Pipeline

Это особенно важно с точки зрения будущих пользователей.

Если завтра кто-то захочет использовать систему только как:

```text
Worldbuilding Database
```

он должен иметь такую возможность.

---

Если другой пользователь захочет использовать её как:

```text
Campaign Encyclopedia
```

тоже.

---

Если третий захочет подключить собственный движок публикации:

```text
Lore Core
    ↓
Custom Exporter
```

он не должен упираться в ограничения архитектуры.

---

# Что бы я сделал следующим этапом

Теперь, когда мы определили границы ядра, я бы начал проектирование не с БД и не с API.

Я бы сначала описал **минимальную онтологию Lore Core**.

То есть ответил на вопрос:

> Какие сущности и связи настолько фундаментальны, что без них не может существовать сама платформа?

Причём не в смысле "Character" или "Location", а именно на уровне абстракций ядра.

Например:

```text
Entity
Entity Type

Relation
Relation Type

Attribute
Attribute Type

Source

Revision

Collection

Taxonomy
```

Мне кажется, что это будет первым действительно архитектурным документом проекта Lore Database / Setting Repository и фактически станет фундаментом всего последующего проектирования.
